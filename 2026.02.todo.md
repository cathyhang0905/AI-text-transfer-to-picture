我的方案可以总结为：**“基于语义角色标注的非侵入式渲染（Semantic Role-Based Non-Invasive Rendering）”**。

它本质上是 **Claude Code (CC) 方案二（标记插入）**与**方案三（两阶段解耦）**的深度融合进化版。

---

### 1. 核心架构：导演与剧组的分工

在这个方案中，AI 不再是“打字员”，而是**“导演”**；代码（Frontend）则是**“剧组”**。

* **AI 的任务（导演）：** 只负责在原文上“贴标签”。它输出一套极其轻量的 **JSON 指令集**，描述每一段文字在视觉上应该扮演什么“角色”。
* **代码的任务（剧组）：** 拿着这份指令，在渲染预览区时，动态地给原文穿上“衣服”（CSS 样式或 React 组件）。

---

### 2. 执行协议（JSON 范例）

AI 不输出任何正文，只输出类似下面的语义索引：

```json
{
  "semantic_map": [
    { "line": 0, "role": "hero_title", "style": "gradient" },
    { "line": 5, "role": "section_divider", "label": "核心逻辑" },
    { "range": [102, 115], "role": "highlight_gold", "text_anchor": "语义单元" }
  ],
  "structural_logic": {
    "list_groups": [[10, 11, 12, 13]],
    "keep_original_first_line": false
  }
}

```

---

### 3. 如何彻底解决你提到的痛点？

#### A. 解决“标题重复” (方案的核心优势)

* **CC 的做法**：试图通过 Prompt 让 AI 别重复写标题，但这在生成式模型中是概率性的。
* **我的做法**：AI 标记 `line: 0` 为 `hero_title`。前端渲染逻辑如下：
* 读取编辑器原文第一行。
* **隐藏**原文第一行的纯文本显示。
* 在原位置插入一个 `<H1>{originalLine0}</H1>` 组件。
* **结果**：内容始终只有一份，物理上杜绝了重复。



#### B. 解决“内容缺失”与“列表断裂”

* **非侵入式渲染**：预览区的基座永远是 `Editor.getValue()`（编辑器里的完整原文）。
* **坐标注入**：代码像“涂鸦”一样，根据 AI 给的行号，在对应行前面插入 Markdown 符号或组件。即使 AI 漏掉了某一行，原文依然完整显示，只是没变色。
* **列表感知**：通过 `list_groups` 索引，前端能感知到 1234 是一个整体，渲染时强制透传 `start` 属性，彻底解决列表重置为 1 的 Bug。

#### C. 解决“理解深度” (超越正则过滤)

* **语义特征评分**：不仅仅看字数，AI 会根据 `role` 判断。
* **例子**：如果 AI 觉得第 8 行虽然长，但是个“金句（Punchline）”，它标记为 `gold_quote`。代码层识别到这个 Role 后，会自动应用一个精美的引言边框，而不是死板地把它变成一个巨大的 H2 标题。

### 4. 为什么这个方案比 CC 的更好？

| 维度 | CC 的方案 (v1-v3) | 我的语义方案 |
| --- | --- | --- |
| **稳定性** | 依赖 AI 完美遵循“不改字”的指令。 | **天然稳定**。代码控制渲染，AI 没机会改字。 |
| **审美上限** | 仅限于标准的 Markdown（标题、高亮）。 | **无限可能**。可以根据语义角色注入任何 UI 组件（卡片、图标、渐变）。 |
| **纠错能力** | AI 提取错了，代码只能通过正则补丁。 | AI 给出意图，代码利用“编辑距离”算法自动对齐最接近的原文。 |
